1What the archive contains

~ 2 minutes of reading

Each .tar.gz produced by pack is a self-contained bundle. The restore process extracts it, verifies every file against the manifest, then restores the payload to the configured locations.

Inside the archiveWhat it is used for
database/cluster.dumpPortable logical restore into ofbiz_world_local.
database/globals.sqlRoles and grants for PostgreSQL.
physical/data/Fast same-version restore of the local PostgreSQL clone.
app/codex-project/The code tree and support scripts from the source machine.
config/env.encEncrypted sync_config.json, decrypted with secrets\env_key.bin.
manifest.jsonSHA-256 integrity list checked before any restore begins.

What the archive does not contain

It does not contain Windows, PostgreSQL installers, Python installers, or Google Drive itself. Those are installed once on the new computer in the next few steps.

Fast restore or portable restore?

Use fast when the new machine matches the source closely: same PostgreSQL major version, same local-clone layout, and same intended restore path. Use portable when in doubt. Portable restore is slower but more forgiving.

Phase A — one-time software installs

3Install PostgreSQL

~ 10 minutes · only once per machine

3a. Download the installer

  1. Open postgresql.org/download/windows.
  2. Download PostgreSQL 18. That matches the source machine and is the best option for fast restore.
  3. Use the x64 Windows installer.

3b. Install it

  1. Run the installer as administrator.
  2. Accept the default installation folder C:\Program Files\PostgreSQL\18.
  3. Keep the server and command-line tools installed.
  4. Set a postgres password and write it down.
  5. Finish the installation.

3c. Verify

"C:\Program Files\PostgreSQL\18\bin\pg_restore.exe" --version

Important for fast restore

Fast restore is designed for the same PostgreSQL major version. If you install a different major version, use portable restore instead.

4Install Python and the dependency

~ 5 minutes · only once per machine
  1. Open python.org/downloads/windows.
  2. Install Python 3 and tick Add python.exe to PATH.
  3. Open cmd in the copied pg_portable_backup folder and run:
py --version
py -3 -m pip install -r requirements.txt

5Install Google Drive for Desktop or rclone

~ 5 minutes · only once per machine

You need a way to reach the archived backups. The current config uses Google Drive for Desktop by default.

  1. Preferred: install Google Drive for Desktop.
  2. Alternative: install rclone and configure a gdrive remote if you plan to switch remote.mode.
  3. Confirm you can browse G:\My Drive\PG_Backups\codex_ofbiz_local.
Phase B — prepare the restore kit

6Copy the pg_portable_backup folder

~ 1 minute

Copy this whole folder onto the new computer. The simplest choice is to put it back in the same location:

C:\Users\PASCA\OneDrive\Documents\Codex\pg_portable_backup

That keeps the launcher scripts and instructions exactly aligned with the paths shown in this guide.

Restore targets are path-sensitive

The app payload restores back to the original absolute path recorded at pack time, which for this system is under C:\Users\PASCA\OneDrive\Documents\Codex\2026-04-19-before-we-start-can-you-access. If the new machine uses a different Windows username or different OneDrive layout, plan for that before doing the real restore.

7Put the encryption key back

~ 1 minute

Copy the saved key file into the new machine's backup folder at:

secrets\env_key.bin

Without it, restore can still recover the database and code, but it will not be able to decrypt the backed-up sync_config.json.

8Check the config on the new machine

~ 3 minutes

Open config.codex.json and confirm it fits the new machine.

Same-machine-style fast restore

For fast restore, keep the same layout as the source machine as closely as possible. The safer fallback is portable restore, which rebuilds from the logical dump.

Path remapping is not automatic

The app folder targets come from the archive metadata. If you intend to restore to a different place than the source machine used, do a dry-run first and read the planned target paths carefully before running the real restore.

Phase C — the actual restore

9See which backups are available

~ 1 minute

You can either use the archive dropdown above or list them from cmd:

py -3 restore.py --config config.codex.json --list

The newest backup appears first. If you do not need a specific date, using latest is fine.

10Run the restore

~ 5-30 minutes · depends on archive size

Recommended order:

  1. Run a dry-run first so the tool verifies the archive and shows you the exact planned targets.
  2. If the machine matches closely and fast restore is intended, use Run Fast Restore.
  3. If you are unsure, use Run Portable Restore.

Manual commands:

py -3 restore.py --config config.codex.json --install latest --restore-mode auto
py -3 restore.py --config config.codex.json --install latest --restore-mode fast
py -3 restore.py --config config.codex.json --install latest --restore-mode portable
py -3 restore.py --config config.codex.json --install latest --drop --restore-mode portable

During the real restore, the tool pauses and asks you to type RESTORE before it overwrites files or loads the database.

What restore does in plain English

  1. Downloads or opens the chosen archive.
  2. Extracts it into staging\restore_<timestamp>.
  3. Verifies every file against manifest.json.
  4. Chooses fast or portable database recovery.
  5. Restores the database.
  6. Takes safety snapshots of any existing app targets before overwriting them.
  7. Restores the app tree to the recorded target path.
  8. Decrypts env.enc back into sync_config.json.

11Verify everything is back

~ 3 minutes

Need to inspect first?

The dry-run mode is designed for that. It does not change anything; it only verifies the archive and prints the planned targets.

Need to recover the pre-restore app state?

The tool makes safety zips in staging\preinstall_<timestamp>\ before overwriting app files. The run log is also saved in logs\.

12Optional: schedule backups on the new machine too

~ 3 minutes

If this machine becomes the new primary machine, create a weekly Windows Task Scheduler job that runs:

/c "C:\Users\PASCA\OneDrive\Documents\Codex\pg_portable_backup\run_backup.bat"

Use the same Task Scheduler approach described in the pack guide. Test the task once manually before relying on it.
