Pack a backup of the local OFBiz PostgreSQL clone

A full step-by-step guide. At the end you will have a single backup file on Google Drive containing the portable logical dump, the fast same-version PostgreSQL snapshot, the code tree, the encrypted sync_config.json, and an integrity manifest.

Phase A — one-time prerequisite install

1Install PostgreSQL command-line tools

~ 10 minutes · only once per machine

The backup uses pg_dump.exe, pg_dumpall.exe, pg_restore.exe, and pg_ctl.exe. They come from the PostgreSQL Windows installer. If PostgreSQL 18 is already installed at C:\Program Files\PostgreSQL\18, you can skip to Step 2.

1a. Download the installer

  1. Open postgresql.org/download/windows.
  2. Click the Windows installer link and download the x64 installer.
  3. Pick PostgreSQL 18. This backup is currently configured for C:\Program Files\PostgreSQL\18\bin.

1b. Run the installer

  1. Right-click the installer and choose Run as administrator.
  2. Accept the default install folder C:\Program Files\PostgreSQL\18.
  3. Keep PostgreSQL Server, pgAdmin 4, and Command Line Tools selected.
  4. You may accept the default data directory. The pack tool does not use that data directory directly; it only needs the binaries.
  5. Set a password for the postgres user. Write it down in case you need to maintain PostgreSQL manually later.
  6. Complete the installation and skip Stack Builder unless you want it.

1c. Verify

Open cmd and run:

"C:\Program Files\PostgreSQL\18\bin\pg_dump.exe" --version

You should see something like pg_dump (PostgreSQL) 18.x.

Already have another PostgreSQL version?

That is acceptable, but then you must update config.codex.json so postgres.bin_dir points at the correct bin folder.

2Install Python and the backup dependency

~ 5 minutes · only once per machine

2a. Install Python 3

  1. Open python.org/downloads/windows.
  2. Download the latest Python 3 installer for Windows.
  3. When the installer starts, tick Add python.exe to PATH.
  4. Click Install Now.

2b. Verify Python

Open a new cmd window and run:

py --version
py -3 -m pip --version

2c. Install the required package

This tool uses the cryptography package to encrypt sync_config.json inside the archive. From this pg_portable_backup folder run:

py -3 -m pip install -r requirements.txt

Pip blocked by proxy or SSL inspection?

Try: py -3 -m pip install --trusted-host pypi.org --trusted-host files.pythonhosted.org -r requirements.txt

3Install Google Drive for Desktop

~ 5 minutes · only once per machine

This is the simplest remote target for this backup. The current config expects the archive to be copied to G:\My Drive\PG_Backups\codex_ofbiz_local.

3a. Download and install

  1. Open Google Drive for Desktop.
  2. Download the installer and sign in with the Google account that should hold the backups.
  3. Let it mount Drive in File Explorer. It is usually the G: drive.

3b. Verify

  1. Open File Explorer.
  2. Confirm that Google Drive is visible and opens.
  3. Browse to G:\My Drive.

4Optional: install rclone

~ 5 minutes · only if you plan to switch remote mode

You can skip this step if you are staying with Google Drive for Desktop. rclone is only needed if you later change remote.mode from gdrive_desktop to rclone.

  1. Open rclone.org/downloads.
  2. Download the Windows build and place rclone.exe somewhere on your PATH.
  3. Run rclone config and create a remote named gdrive if you want to use the config default.

Recommended default

For this system, staying on Google Drive for Desktop is simpler. It is already the configured default and matches the pack/unpack flow on these pages.

Phase B — the actual backup setup and run

5Check the config file

~ 3 minutes

Open config.codex.json in Notepad or VS Code and confirm these values are still correct for this machine:

SettingExpected valueWhy it matters
postgres.bin_dirC:\Program Files\PostgreSQL\18\binWhere pg_dump and pg_restore come from.
postgres.host / port127.0.0.1 / 55432The local OFBiz clone listens there.
postgres.databaseofbiz_world_localThe logical dump targets this database.
local_cluster.data_dirC:\Users\PASCA\OneDrive\Documents\Codex\scripts\.local-postgres\dataThe fast physical snapshot comes from here.
sources[0].pathC:\Users\PASCA\OneDrive\Documents\Codex\scriptsThe code tree and scripts are copied from here.
env_fileC:\Users\PASCA\OneDrive\Documents\Codex\scripts\sync_config.jsonThis file is encrypted into the archive.
remote.gdrive_desktop_pathG:\My Drive\PG_Backups\codex_ofbiz_localWhere the final archive is uploaded.

What gets backed up?

The archive contains two database recovery forms: a portable logical dump and a fast physical snapshot of the local clone. It also includes the OFBiz code folder, operator scripts, and an encrypted copy of sync_config.json.

About the local clone

The pack command can start the local PostgreSQL clone automatically if it is down. That clone lives under .local-postgres and is excluded from the ordinary code copy because it is already captured separately as the fast snapshot payload.

6Make sure the Drive destination exists

~ 1 minute
  1. Open File Explorer.
  2. Browse to G:\My Drive\PG_Backups.
  3. If codex_ofbiz_local does not exist yet, create it.

The real backup run copies the final .tar.gz file into that folder. If the folder does not exist, upload will fail.

7Run a dry-run pack first

~ 3-10 minutes

This validates the whole flow without uploading anything. It is the safe first test after changing config or moving the tool to another machine.

You can also run it manually from cmd:

py -3 backup.py --config config.codex.json --dry-run

Expected progress lines include:

step 1/7: pg_dump -Fc ...
step 2/7: pg_dumpall --globals-only ...
step 3/8: capturing fast-recovery cluster snapshot ...
step 4/8: copying source trees ...
step 5/8: encrypting .env ...
step 6/8: building manifest ...
step 7/8: creating tar.gz ...
step 8/8: DRY-RUN - skipping upload + prune

If the database clone is not running

The backup now preflights the local cluster. If port 127.0.0.1:55432 is down, it will try to start the local clone before running pg_dump.

8Run the real backup

~ 5-30 minutes · depends on archive size

Now run the full pack. This performs the same local work as the dry-run and then uploads the final archive to Google Drive and prunes older remote copies beyond retention_weeks.

Manual command:

py -3 backup.py --config config.codex.json

Or simply double-click run_backup.bat.

Wait for the final line === DONE OK ===. Large archives can sit for a while on the creating tar.gz step because compression takes time.

9Verify the archive reached Google Drive

~ 1 minute
  1. Open G:\My Drive\PG_Backups\codex_ofbiz_local.
  2. Look for a file named like codex_ofbiz_local_20260423T061922.tar.gz.
  3. Open drive.google.com and confirm the same file appears after Drive sync finishes.
Phase C — optional automation and recovery readiness

10Optional: schedule weekly backups in Windows

~ 3 minutes

This package does not ship a dedicated scheduler script, but Windows Task Scheduler works well with run_backup.bat.

  1. Open Task Scheduler.
  2. Choose Create Basic Task.
  3. Name it something like pg_portable_backup_codex.
  4. Choose Weekly, then pick the day and time you want.
  5. Choose Start a Program.
  6. Program/script: cmd.exe
  7. Add arguments:
/c "C:\Users\PASCA\OneDrive\Documents\Codex\pg_portable_backup\run_backup.bat"
  1. Start in: C:\Users\PASCA\OneDrive\Documents\Codex\pg_portable_backup
  2. Finish the wizard and run the task once manually to verify it works.

Best time to schedule

Pick a quiet time when the machine is on, Google Drive is signed in, and the local clone is not being actively used.

11Store the encryption key safely critical

~ 1 minute

The archive stores sync_config.json in encrypted form. The key to decrypt it is kept locally at:

secrets\env_key.bin

Without that file, unpack can restore the database and code, but it cannot recreate the encrypted config payload.

  • Copy env_key.bin to a USB stick in a safe place, or
  • store it in a secure password manager as an attachment, or
  • save it in another protected location that is not the same Google Drive folder as the archives.

Do not store the key next to the backups

If someone gets both the archive and env_key.bin, they get the secrets too. Keep them separate on purpose.

You are done.

The backup package is now both usable and recoverable: it can make a fresh archive on demand, it can be scheduled, and the decrypt key is preserved for future unpack on another computer.
