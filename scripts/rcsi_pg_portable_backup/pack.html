Pack a backup of the PostgreSQL database

A printable, step-by-step guide for anyone. You will end up with a single file on Google Drive that contains everything needed to rebuild the system on another computer: the database, both application folders, the encrypted .env, and a manifest for integrity checking.

What you are about to do

Steps 1-4 are one-time installation. Steps 5-8 are the actual backup. Steps 9-10 only run once, the first time.

  1. Install PostgreSQL (one-time)
  2. Install Python 3.10+ and dependencies (one-time)
  3. Install Google Drive for Desktop (one-time)
  4. (Optional) Install rclone (one-time)
  5. Open the config file and look at it
  6. Make sure the Drive destination folder exists
  7. Run a test backup (dry-run)
  8. Run the real backup
  9. Verify the file appeared on Google Drive
  10. Turn on automatic weekly backups (one-time)
  11. Save the encryption key in a safe place (one-time)
Phase A — one-time prerequisite install

1Install PostgreSQL

~ 10 minutes · only once per machine

The backup uses the tools pg_dump.exe and pg_dumpall.exe that ship with the PostgreSQL installer. If PostgreSQL is already installed on this computer, you can skip to Step 2.

1a. Download the installer

  1. Go to postgresql.org/download/windows and click Download the installer (EDB's interactive installer).
  2. Pick version 18 (or newer). That is what this system is configured for in config.rcsi.json.
  3. Download the x64 Windows .exe.

1b. Run the installer

  1. Right-click the .exe and choose Run as administrator.
  2. Accept the default install location: C:\Program Files\PostgreSQL\18.
  3. On the Select Components screen, keep PostgreSQL Server, pgAdmin 4, and Command Line Tools ticked. Stack Builder is optional and can be unticked.
  4. Accept the default data directory C:\Program Files\PostgreSQL\18\data.
  5. Set a password for the postgres superuser. Write it down. For this system it is comm1985, already in config.rcsi.json.
  6. Accept the default port 5432.
  7. Accept the default locale, click Next, and let it install (~5 minutes).
  8. You can tick out of Stack Builder at the end — it's not needed.

1c. Verify

Open cmd and run:

"C:\Program Files\PostgreSQL\18\bin\pg_dump.exe" --version

You should see something like pg_dump (PostgreSQL) 18.x.

Already have a different PostgreSQL version installed?

That's fine — just point config.rcsi.json → postgres.bin_dir at the right version folder (for example C:\\Program Files\\PostgreSQL\\17\\bin). The backup can run with 15, 16, 17, or 18.

2Install Python 3.10+ and the dependencies

~ 5 minutes · only once per machine

2a. Download and install Python

  1. Go to python.org/downloads/windows and download the latest Python 3.12 (or 3.10 / 3.11 / 3.13).
  2. Run the installer. On the very first screen, tick the box "Add python.exe to PATH". This is important.
  3. Click Install Now.

2b. Verify

Open a new cmd window (a new one, so it picks up the PATH change) and run:

py --version
py -3 -m pip --version

Both should print a version number.

2c. Install the one Python package the backup needs

The only dependency is cryptography (used to encrypt the .env file). Open cmd inside this pg_portable_backup folder and run:

py -3 -m pip install -r requirements.txt

Corporate network / proxy?

If pip install fails because of SSL or proxy issues, try: py -3 -m pip install --trusted-host pypi.org --trusted-host files.pythonhosted.org -r requirements.txt

3Install Google Drive for Desktop

~ 5 minutes · only once per machine

This is the simplest way to push backups to the cloud: Google Drive for Desktop maps your Drive as a drive letter (usually G:), and copying a file into that drive letter uploads it.

3a. Download and install

  1. Go to google.com/drive/download → Download Drive for Desktop.
  2. Run the installer and accept defaults.
  3. When it asks you to sign in, sign in with the Google account where the backups should live.

3b. Confirm the drive letter

  1. Open File Explorer. You should see a new drive called Google Drive (G:) (or a different letter — note whichever letter you see).
  2. Inside G:\ you will see a folder called My Drive that mirrors drive.google.com.

Tip: stream vs. mirror

In Drive for Desktop's preferences, the default is "Stream files" — files live in the cloud until you open them. That's fine for backups: we only write into the folder, never read from it routinely.

4(Optional) Install rclone as a fallback

~ 5 minutes · only if you need it

Skip this step unless you plan to switch remote.mode to rclone (for example, to run on a server without a desktop session). Google Drive for Desktop is simpler and works fine for most users.

4a. Download rclone

  1. Go to rclone.org/downloads and download the Windows AMD64 zip.
  2. Unzip it. Inside is a single rclone.exe.
  3. Put rclone.exe somewhere on your PATH, e.g. C:\Tools\rclone\rclone.exe, and add that folder to PATH.

4b. Configure a Google Drive remote

Open cmd and run:

rclone config

Interactive prompts:

  • n for new remote.
  • Name: gdrive (or whatever matches the config's rclone_remote prefix).
  • Storage: pick Google Drive.
  • Leave client ID / secret blank (use rclone's defaults).
  • Scope: drive (full access).
  • Say yes to auto config; a browser opens, sign in, grant access.
  • Say no to "Configure this as team drive".
  • Confirm, then q to quit.

Verify:

rclone lsd gdrive:

You should see your Drive's top-level folders listed.

Phase B — the actual backup

5Open the config file and glance at it

~ 2 minutes

Open config.rcsi.json in Notepad (or any editor). You don't need to change anything if you are using the system that was set up for you. The key lines:

"postgres": {
    "bin_dir":  "C:\\Program Files\\PostgreSQL\\18\\bin",
    "database": "rcsidatabase",
    "user":     "postgres",
    "password": "comm1985",
    ...
},
"sources": [
    { "label": "rcsi-catalog-export",
      "path":  "C:\\Users\\PASCA\\OneDrive\\Documents\\Scripts\\RCSi - Catalog Export" },
    { "label": "erp-scripts",
      "path":  "C:\\Users\\PASCA\\OneDrive\\Documents\\Scripts\\ERP" }
],
"remote": {
    "mode": "gdrive_desktop",
    "gdrive_desktop_path": "G:\\My Drive\\PG_Backups\\rcsi_erp"
}

What is in the archive?

Each backup bundles the full PostgreSQL database dump, both application folders listed under sources (RCSi - Catalog Export and ERP), the encrypted .env file, and an integrity manifest (SHA-256 for every file). When you restore, each source goes back to the exact absolute path it came from.

Backslashes in JSON

Windows paths need to be written with double backslashes inside JSON strings: "C:\\Program Files\\...", never just "C:\Program Files\...". If the backup errors out with a JSON parse error, that's almost always the cause.

6Make sure the Google Drive destination folder exists

~ 1 minute
  1. Open File Explorer.
  2. Go to G:\My Drive (or whichever letter your Drive uses).
  3. Create the folder structure from the config, for example: G:\My Drive\PG_Backups\rcsi_erp. If it is already there, great.

Tip: wrong drive letter

If Google Drive uses a different letter on your machine (e.g. H:), either change the letter in the Drive for Desktop settings, or edit the config's gdrive_desktop_path to match.

7Run a test backup (dry-run)

~ 2 minutes · no files are uploaded

A dry-run builds the backup on your hard drive only, so we can check that everything works before sending anything to Drive.

  1. Open cmd in this folder (Shift + right-click the folder in File Explorer → "Open in Terminal" or "Open command window here").
  2. Run:
py -3 backup.py --dry-run

You should see lines like step 1/7: pg_dump -Fc ... and finish with === DONE OK ===. The finished .tar.gz file will be inside the staging\ subfolder.

If it fails

Open the newest file in the logs\ subfolder. The last line usually tells you exactly what went wrong:

  • password authentication failed → fix postgres.password in the config.
  • No such file or directory .../bin/pg_dump → fix postgres.bin_dir.
  • source not found: ... → fix the sources[].path for that label.
  • Expecting property name enclosed in double quotes → JSON syntax error, usually a missing " or an unescaped backslash.

8Run the real backup

~ 5-30 minutes · depends on DB size

Now the real thing. It will do everything the dry-run did, plus copy the archive into your Google Drive folder and delete older archives past the retention limit (retention_weeks in the config).

Easy way: double-click run_backup.bat in File Explorer.

Or, from cmd:

py -3 backup.py

You will see the same progress lines, plus a final uploaded: G:\My Drive\PG_Backups\... line.

9Verify the backup reached Google Drive

~ 1 minute
  1. Open File Explorer → G:\My Drive\PG_Backups\rcsi_erp.
  2. You should see a file named like rcsi_erp_20260421T094512.tar.gz (the numbers are the year, month, day, then T, then time).
  3. Open drive.google.com in your browser. After a moment (Drive has to sync) the same file appears under the same folder. That is your offsite copy.
Phase C — one-time finishing touches

10Turn on weekly automatic backups

~ 1 minute · you only do this once

This step makes Windows run the backup for you every week. You can pick the day and the time.

  1. Open PowerShell as Administrator (Start menu → type "powershell" → right-click → Run as administrator).
  2. Navigate to this folder and register the task:
cd "C:\Users\PASCA\OneDrive\Documents\Scripts\RCSi - Catalog Export\pg_portable_backup"
Set-ExecutionPolicy -Scope Process Bypass
.\register_task.ps1 -Day Sunday -Time 03:00

You can change -Day and -Time to whatever you prefer. Pick a time when the computer is on and the database is quiet (early morning is a safe default).

How do I know it worked?

In PowerShell, run Get-ScheduledTask -TaskName pg_portable_backup. You should see status Ready. To test it now: Start-ScheduledTask -TaskName pg_portable_backup.

Uninstall or change schedule

To change the schedule, just run register_task.ps1 again with new flags — it will overwrite. To remove: .\register_task.ps1 -Uninstall.

11Save the encryption key in a safe place critical

~ 1 minute

The .env file (containing passwords and secrets) was encrypted. The key that can decrypt it is stored only on this computer at:

secrets\env_key.bin

Without that key, the .env inside future backups will be unreadable. Do one of:

  • Copy env_key.bin onto a USB stick and put it in a drawer.
  • Upload it (once) to a password manager like Bitwarden or 1Password as a "secure file" attachment.
  • Email it to yourself using a secure channel, clearly labeled.

Do NOT store env_key.bin in the same Google Drive folder as the backups.

If both live in the same place, whoever gets access to that folder gets access to the passwords too. The point of separating them is that someone would need both to read your secrets.

You are done.

From now on, Windows will make a backup every week by itself. Come back here any time you want to change the schedule or make an ad-hoc backup. If this machine ever dies, use unpack.html on the new machine to get everything back.
