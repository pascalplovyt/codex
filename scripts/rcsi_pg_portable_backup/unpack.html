Unpack a backup onto a new computer

A step-by-step guide for anyone. At the end, the new computer will have the same database and the same two application folders (RCSi - Catalog Export and ERP) as the machine where the backup was made.

1What the archive contains

~ 2 minutes of reading

Each .tar.gz produced by pack is a self-contained bundle. When you unpack it with restore.py, everything lands at its original location:

Inside the archiveIs restored to
database/cluster.dump Loaded into PostgreSQL on the new machine (database name from the config).
database/globals.sql Roles & grants applied to the new cluster.
app/rcsi-catalog-export/ C:\Users\<you>\OneDrive\Documents\Scripts\RCSi - Catalog Export\ (the exact absolute path recorded when packing, from _sources.json).
app/erp-scripts/ C:\Users\<you>\OneDrive\Documents\Scripts\ERP\
config/env.enc Decrypted with secrets/env_key.bin, written to the env_file path in the config.
manifest.json SHA-256 of every file — verified before any restore starts.

What the archive does NOT contain

It does not include the PostgreSQL server binaries, Python, or Google Drive for Desktop. Those are software installs you do once on the new machine (Steps 3-5 below). The archive only contains your data and your code.

2Three things need to be on the new computer

~ 1 minute
ItemWhyWhere it comes from
Backup archive
rcsi_erp_YYYYMMDDThhmmss.tar.gz		 Contains the database + both code folders + encrypted .env. Google Drive folder where weekly backups land.
The pg_portable_backup folder (the one this page is inside) The unpack program itself. Copy it from the old computer, or from any sync / copy of it.
Encryption key
secrets\env_key.bin		 Unlocks the encrypted .env inside the archive. You saved it in the last step of the pack guide — retrieve it from the USB stick / password manager you used.

Why three things?

The archive alone is enough to rebuild the database and the applications, but to make them fully functional we also need the program that does the unpacking, and the key that decrypts the secrets file. Each one on its own is useless, which is why storing them separately is safe.

Phase A — one-time software installs

3Install PostgreSQL

~ 10 minutes · only once per machine

3a. Download the installer

  1. Go to postgresql.org/download/windows and click Download the installer (EDB's interactive installer).
  2. Pick the same major version as the original machine, or newer. The original machine for this system runs PostgreSQL 18, so install 18 or newer.
  3. Download the x64 Windows .exe.

3b. Run the installer

  1. Right-click the .exe and choose Run as administrator.
  2. Accept the default install location: C:\Program Files\PostgreSQL\18.
  3. On the Select Components screen, keep PostgreSQL Server, pgAdmin 4, and Command Line Tools ticked. Stack Builder can be unticked.
  4. Accept the default data directory C:\Program Files\PostgreSQL\18\data.
  5. Set a password for the postgres superuser. Write it down. You will type it into config.rcsi.json in Step 8.
  6. Accept the default port 5432.
  7. Accept the default locale, click Next, and let it install.
  8. Skip Stack Builder at the end.

3c. Verify

Open cmd and run:

"C:\Program Files\PostgreSQL\18\bin\pg_restore.exe" --version

You should see pg_restore (PostgreSQL) 18.x.

Older / different major version

If you must use an older PostgreSQL (say 15 or 16), you can still restore most data — but features that require the newer version may fail. Best practice: install the same major version the source used (18), or newer.

4Install Python 3.10+ and dependencies

~ 5 minutes · only once per machine

4a. Download and install Python

  1. Go to python.org/downloads/windows and grab Python 3.12 (3.10 / 3.11 / 3.13 also fine).
  2. Run the installer. On the very first screen, tick the box "Add python.exe to PATH". Do not skip this.
  3. Click Install Now.

4b. Verify

Open a new cmd window (a new one, so it picks up the PATH change) and run:

py --version
py -3 -m pip --version

4c. Install the one required package

You will do this after you copy the pg_portable_backup folder onto this machine in Step 6, but the command to run then is:

py -3 -m pip install -r requirements.txt

(The only dep is cryptography, used to decrypt the .env file.)

5Install Google Drive for Desktop (or rclone)

~ 5 minutes · only once per machine

Pick one of the two options below. It must match the remote.mode set in config.rcsi.json. The default is gdrive_desktop.

Option A — Google Drive for Desktop (default, simplest)

  1. Go to google.com/drive/download → Download Drive for Desktop.
  2. Run the installer, accept defaults.
  3. Sign in with the Google account that holds the backups.
  4. Wait until you see a drive letter (usually G:\) in File Explorer.
  5. Navigate to G:\My Drive\PG_Backups\rcsi_erp and confirm the .tar.gz files are visible.

File not downloading?

In streaming mode, a file in Drive is only a placeholder until you open it. If a file has a little cloud icon, right-click → Offline access → Available offline to pull the bytes before you feed it to restore.py. Or let restore.py trigger the download for you — it will still work, just a bit slower the first time.

Option B — rclone

  1. Download rclone.exe from rclone.org/downloads (Windows AMD64 zip).
  2. Unzip and place rclone.exe somewhere on PATH, e.g. C:\Tools\rclone\.
  3. Run rclone config and create a new remote of type Google Drive, named to match the config's rclone_remote prefix (default: gdrive). Sign in through the browser when prompted.
  4. Verify: rclone lsd gdrive:PG_Backups/rcsi_erp lists the archives.
  5. In config.rcsi.json, set remote.mode to "rclone".
Phase B — move the restore tool into place

6Copy the pg_portable_backup folder onto the new computer

~ 2 minutes

On the original machine this folder is at C:\Users\<user>\OneDrive\Documents\Scripts\RCSi - Catalog Export\pg_portable_backup. Copy the whole folder to the new machine. The exact path doesn't matter for the restore itself, but something like C:\pg_portable_backup keeps things simple.

Then open cmd inside the folder and install the Python dependency (see Step 4):

py -3 -m pip install -r requirements.txt

What about the staging\ and logs\ folders?

Those are scratch folders and are safe to copy or leave empty — the scripts recreate them on first run. Do not copy secrets\env_key.bin via any unsecured channel (see Step 7 for how to bring that file in safely).

OneDrive path on the new machine

If the new machine uses a different Windows user name, the path C:\Users\<user>\OneDrive\... will be different. The restore uses the path that was recorded at pack time (from app/_sources.json) — if that path does not exist on the new machine, it will be created. You may want to update the config.rcsi.json and the original application's own paths afterwards.

7Put the encryption key back in place critical

~ 1 minute

Retrieve env_key.bin from wherever you stored it (USB stick, password manager, etc.) and place it exactly here on the new machine:

pg_portable_backup\secrets\env_key.bin

If you can't find the key

The restore will still succeed for the database and app files, but the .env secrets file will not be decrypted. You'll have to recreate it by hand afterwards. Skip the key copy and just add --skip-env when you run the restore.

8Open the config and check a few values

~ 2 minutes

Open config.rcsi.json in Notepad. Adjust if necessary:

  1. postgres.password — set it to the password you chose when installing PostgreSQL in Step 3.
  2. postgres.bin_dir — set it to the folder that holds pg_restore.exe, typically C:\\Program Files\\PostgreSQL\\18\\bin. Use double backslashes inside JSON.
  3. postgres.database — leave as rcsidatabase unless the receiving system uses a different name.
  4. remote.gdrive_desktop_path — point it at the folder on the new computer where Google Drive for Desktop shows the backups (the drive letter may differ on the new machine).
  5. env_file — this is where the decrypted .env will be written. Default: C:\\Users\\<you>\\OneDrive\\Documents\\Scripts\\RCSi - Catalog Export\\rcsidatabase\\.env.

You don't need to list the source paths again

Each source's absolute path on the original machine was baked into app/_sources.json at pack time. The restore reads that file and puts each source back at that exact path.

Phase C — the actual restore

9See what backups are available

~ 1 minute

In the pg_portable_backup folder, open cmd and run:

py -3 restore.py --list

You should see a table like:

timestamp          filename
-----------------  --------------------------------------------
20260421T030000    rcsi_erp_20260421T030000.tar.gz
20260414T030000    rcsi_erp_20260414T030000.tar.gz
20260407T030000    rcsi_erp_20260407T030000.tar.gz
...

The newest is at the top. If you're happy with it, continue to the next step. Otherwise note the timestamp of the one you want.

10Run the restore

~ 5-30 minutes · depends on DB size

Easy way: double-click run_restore.bat. It will show the list, ask which timestamp to install, and run.

Or from cmd, to use the newest:

py -3 restore.py --install latest

Or to pick a specific date:

py -3 restore.py --install 20260414T030000

You will be asked to type the word RESTORE in capital letters to confirm — this is an intentional safety gate, because the next step overwrites application files and loads the database.

If the target database already exists with data

By default the restore adds into the existing database. If you want to drop it first and start clean, add --drop:

py -3 restore.py --install latest --drop

What the restore is doing, in plain English

  1. Downloads the archive from Google Drive into staging\.
  2. Extracts it and checks every file against the SHA-256 list inside. If any file is corrupt, it stops.
  3. Applies the globals.sql (roles and grants) to the new Postgres, ignoring harmless "role already exists" errors.
  4. Runs pg_restore to load the database dump into the database named by postgres.database in the config.
  5. For each labelled source in the archive (rcsi-catalog-export, erp-scripts): takes a safety snapshot of the current folder at its destination into staging\preinstall_<ts>\<label>.zip, then copies the archived version on top.
  6. Decrypts env.enc → .env at the original path, using secrets\env_key.bin.

11Verify it worked

~ 3 minutes

Something looks wrong?

Nothing was thrown away. The pre-restore state of each source folder is zipped up at staging\preinstall_<timestamp>\<label>.zip — unzip it and you're back where you started. Database rollback is easier still: re-run restore with --drop and pick a different archive.

Log files for every run live in the logs\ folder.

12(Optional) Schedule weekly backups here too

~ 1 minute

If this new machine is going to be the primary one from now on, set up weekly backups the same way the old machine had them. Open PowerShell as Administrator in this folder and run:

Set-ExecutionPolicy -Scope Process Bypass
.\register_task.ps1 -Day Sunday -Time 03:00

Details are in pack.html, Step 10.

You are done.

The new machine now holds an exact copy of the database and both application folders from the source machine. From here on, life proceeds normally — and if you enabled Step 12, next Sunday at 03:00 this machine will write its own fresh backup into Google Drive.
